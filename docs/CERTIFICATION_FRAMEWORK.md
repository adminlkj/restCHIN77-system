# إطار اعتماد دورات العمل (Business Process Certification Framework)

> هذا الإطار يحكم منهجية اعتماد أي دورة عمل في النظام قبل الإنتاج.
> الاعتماد يعني: "دورة العمل هذه مُختبَرة بالكامل من البداية للنهاية،
> وتتبع أثرها عبر كل الكيانات والتقارير، وتُطابق ميزان المراجعة."

---

## 1. المبادئ الحاكمة

### 1.1 الاعتماد بالدورة، لا بالشاشة
الوحدة الأساسية للاعتماد هي **دورة العمل الكاملة** (مثلاً: دورة البيع)،
وليس الشاشة. اختبار شاشة POS وحدها لا يعتمد دورة البيع — يجب تتبّع الأثر
من اختيار الطاولة إلى ظهور الرصيد في ميزان المراجعة.

### 1.2 المصدر الموحّد للحقيقة المالية
كل رقم مالي يُستخرج من **قيود اليومية المرحّلة** (`isPosted=true`).
لا يجوز قراءة الأرقام المالية من سجلات العمليات (فواتير، سندات) مباشرةً —
هذا يُنشئ مصادر متعددة غير قابلة للمطابقة.

### 1.3 منع الإصلاحات العشوائية
أي إصلاح داخل دورة معتمدة يُلغي اعتمادها. يجب إعادة اختبار كامل للدورة
قبل الدمج. الاعتماد لا يبقى صالحاً بعد أي تعديل بنيوي.

### 1.4 لا اعتماد بدون اختبار حيّ
التحليل النظري للكود لا يكفي. الاعتماد يتطلب **تشغيلاً فعلياً** بقاعدة
بيانات حيّة وتتبّع أثر العمليات في الجداول، لا في الكود فقط.

---

## 2. مراحل الاعتماد الست

كل دورة تمرّ بهذه المراحل بالترتيب. لا تُعتبر الدورة معتمدة إلا باجتياز كل المراحل.

| المرحلة | الاسم | ما يتم التحقق منه |
|---|---|---|
| **M1** | الواجهة | كل زر/حقل/قائمة/نافذة/رسالة/اختصار/إخفاء |
| **M2** | قواعد العمل | كل حالة استخدام كـ Test Case مستقل |
| **M3** | المحركات | القيد متوازن + المخزون + الذمم + الضريبة + الإيراد + التكلفة |
| **M4** | قاعدة البيانات | SalesInvoice/JE/Inventory/Balance متطابقة بعد كل عملية |
| **M5** | اللوغات | لا أخطاء صامتة + لا race conditions + لا طلبات مكررة |
| **M6** | Regression | إعادة اختبار كامل بعد أي إصلاح |

### معيار النجاح لكل مرحلة
- **PASS**: كل Test Case يجتاز، أو الفجوة موثَّقة كقرار منتج مقبول.
- **FAIL**: أي Test Case حرج يفشل. يلزم الإصلاح ثم إعادة M1-M6.

---

## 3. تتبع الأثر (End-to-End Tracing)

لكل دورة، نرسم سلسلة الكيانات التي تُكتب/تُقرأ، ونتحقق أن كل حلقة تُطابق التالية:

```
[العملية] → [الكيان 1] → [الكيان 2] → ... → [التقرير النهائي] → [ميزان المراجعة]
```

### قاعدة المطابقة الذهبية
بعد أي عملية مالية، يجب أن تتطابق:
1. إجمالي المبلغ على المستند التشغيلي (SalesInvoice.totalAmount)
2. إجمالي القيد المحاسبي (JE.totalDebit == JE.totalCredit)
3. نفس الرقم يظهر في ميزان المراجعة على الحساب الصحيح
4. نفس الرقم يظهر في التقرير المتخصص (VATReport/DailySales/Clients)

**أي اختلاف = فشل الاعتماد.**

## 4. شهادة الاعتماد

كل دورة معتمدة تُمنح شهادة بالشكل:

```
╔══════════════════════════════════════════════════════════╗
║          [DORA] CERTIFICATION                            ║
╠══════════════════════════════════════════════════════════╣
║  UI (M1) ........................ ✅ PASS                ║
║  Workflow (M2) ................... ✅ PASS               ║
║  Engines (M3) .................... ✅ PASS               ║
║  Database (M4) ................... ✅ PASS               ║
║  Logs (M5) ....................... ✅ PASS               ║
║  Regression (M6) ................. ✅ PASS               ║
║  E2E Reconciliation .............. ✅ PASS               ║
║  Coverage ........................ N%                    ║
║                                                          ║
║  Status .......................... ✅ CERTIFIED          ║
╚══════════════════════════════════════════════════════════╝
```

**القاعدة:** لا يُسمح بالانتقال لدورة تالية إلا باجتياز كل المراحل.
التغطية المستهدفة ≥ 90%.

## 5. الدورات المطلوب اعتمادها

| # | الدورة | الشاشات المتضمنة | الحالة |
|---|---|---|---|
| 1 | **دورة البيع** (Sales Cycle) | POS, Tables, SalesInvoices, ClientPayments, Receipts printing | قيد التطوير |
| 2 | **دورة المشتريات** (Procurement) | PurchaseOrders, GoodsReceipts, SupplierInvoices, SupplierPayments | مؤجلة |
| 3 | **دورة المخزون** (Inventory) | Inventory, Warehouses, StockMovements, Menu | مؤجلة |
| 4 | **دورة الموارد البشرية** (HR) | Employees, PayrollRuns, Attendance, Advances | مؤجلة |
| 5 | **دورة المحاسبة** (Accounting) | ChartAccounts, JournalEntries, TrialBalance, Reports, VAT | مؤجلة |
| 6 | **دورة النقدية** (Cash) | BusinessDay, Z-Report, DailySalesReport | مؤجلة |

كل دورة تُعتمد بترتيب الأولوية للأعمال (المبيعات أولاً).

## 6. توثيق الفجوات

كل قدرة مفقودة تُسجَّل بدقة:
- **PRESENT**: منفّذة وقابلة للاختبار.
- **PARTIAL**: منفّذة جزئياً، تحتاج إكمال.
- **MISSING**: غير منفّذة — إما تُضاف قبل الاعتماد، أو تُوثَّق كقرار منتج مُؤجَّل.

الفجوات MISSING لا تفشل الاعتماد إذا كانت موثَّقة ومقبولة من أصحاب القرار،
لكنها تُسجَّل في "خريطة طريق ما بعد الاعتماد".

---

## 7. متطلبات الاختبار الحيّ

لاعتماد دورة بشكل صحيح، يجب:

1. **بيئة اختبار** بقاعدة بيانات نظيفة + دليل حسابات قياسي + عملاء/موردون تجريبيون.
2. **سكربت بذور** (seed script) يُنشئ:
   - فرعاً تجريبياً + طاولات + قائمة طعام.
   - مستخدمين بأدوار مختلفة (مالك، مدير، كاشير).
3. **تتبّع أثر كل عملية** عبر:
   - فحص `SalesInvoice` بعد البيع.
   - فحص `JournalEntry` المطابق.
   - فحص ميزان المراجعة يطابق.
   - فحص التقرير اليومي/VAT يطابق.
4. **اختبار حدود** (edge cases): سلة فارغة، خصم 100%، دفع زائد، بيع بعد منتصف الليل.
5. **اختبار أمني**: كاشير يحاول فتح فرع غير مصرّح، إلغاء بكلمة مرور خاطئة.
6. **اختبار تعدد أجهزة**: كاشيران على نفس الفرع، مسودة من جهاز تظهر على الآخر.

---

## 8. الحوكمة

- **الاعتماد قرار جماعي**: مطوّر + مالك المنتج + محاسب.
- **لا merge قبل الاعتماد**: أي PR يمسّ دورة معتمدة يحتاج إعادة اعتماد.
- **سجل الاعتماد**: كل شهادة تُؤرشف بتاريخ + نسخة الكود + نتائج الاختبارات.
- **مراجعة دورية**: إعادة اعتماد كل ربع سنوياً أو بعد تغييرات بنيوية كبيرة.

---

## ملاحظة صدق

هذا الإطار يحدّد **المعيار المطلوب** للاعتماد. تنفيذه الفعلي يتطلب:
- بيئة اختبار حيّة (غير متاحة في بيئة التطوير الحالية بلا PostgreSQL).
- تنفيذ مصفوفات Test Cases لكل دورة.
- تتبّع النتائج وتوثيقها.

الاعتماد الحقيقي = اختبار حيّ، لا تحليل كود. التحليل النظري يمهّد للاختبار
لكنه لا يُغني عنه.
